# Reverie Terminal Strategy Spike

> **Historical, superseded.** This is the original Ghostty/`libghostty` research and strategy spike. The terminal was rebuilt from scratch since; the canonical, current design is [`terminal/`](terminal/README.md). Parts of this doc (transcript capture, deep-history replay, in-terminal find, benchmark proofs) describe a model the rebuild removed. Kept for history only; do not build from it.

## Decision summary

For v1, Reverie should treat a Ghostty-quality terminal experience as a core product requirement. The product exists in large part because existing agent-session organizers make the terminal feel slow, brittle, or unpleasant; we cannot ship a v1 that repeats that failure.

Recommended path:

1. Build a terminal abstraction around sessions, PTYs, rendering, input, resize, copy/paste, and process lifecycle.
2. Pursue Ghostty-backed rendering as the preferred v1 path using `libghostty-vt` / Rust bindings, with an isolated spike before binding it to the main app shell.
3. Keep fallback thinking only as architectural insurance, not as permission to ship a weak terminal experience.

The product architecture should not depend directly on any one terminal implementation. Reverie's core value is organizing and restoring agent sessions through a terminal that feels fast and native; the terminal layer is critical product surface, and the abstraction exists to protect quality and implementation flexibility rather than lower the bar.

## Findings

### Ghostty / libghostty

Ghostty is a high-quality cross-platform terminal written primarily in Zig, with platform-specific app layers. The project exposes a C-compatible embeddable library story, but the currently split-out component most relevant to us is `libghostty-vt`.

`libghostty-vt` provides terminal emulation primitives:

- escape sequence parsing
- terminal state
- scrollback
- wrapping and resize reflow
- input encoding
- render state / incremental render updates
- formatting utilities

The important limitation: it is not a complete desktop terminal widget. We still need to own or integrate:

- PTY creation and process spawning
- input routing
- resize handling
- font shaping/rendering
- GPU/canvas/webview rendering
- selection, copy, paste
- terminal theming
- cross-platform packaging/build behavior

The public docs also warn that the API is still in development, not stable, and breaking changes should be expected.

### Rust bindings

The available Rust path is currently third-party:

- `libghostty-vt` crate: safe Rust API for `libghostty-vt`
- `libghostty-vt-sys`: raw FFI bindings
- repository: `uzaaft/libghostty-rs`

Useful facts:

- current crate version observed: `0.1.1`
- Rust version requirement observed: `1.93`
- build requires Zig `0.15.x` on PATH
- by default, the build fetches Ghostty source from a pinned commit
- can use `GHOSTTY_SOURCE_DIR` to point at a local Ghostty checkout
- pre-1.0 and tracks a pinned Ghostty source revision

This is promising, but it adds build-chain risk to a Tauri v2 app: Rust + Tauri + platform webview + Zig + Ghostty source pinning.

API details worth preserving for the spike:

- Core safe-wrapper types include `Terminal`, `TerminalOptions`, and `RenderState`.
- Terminal state is render-state driven; Ghostty gives us the visible terminal cells/rows to draw, not a ready-made Tauri widget.
- Input helpers exist for encoding keys, mouse events, focus events, and paste behavior.
- `fmt` helpers can emit terminal content as plain text, VT, or HTML, which may help debugging but should not be assumed to be the production renderer.
- Public Rust docs are still partial, and the safe wrapper objects are not `Send`/`Sync`; the renderer state likely needs to live on a dedicated single-threaded owner with message passing from the PTY/runtime.

### ratatui-ghostty

`ratatui-ghostty` exists and is useful evidence that Ghostty's VT core can be rendered through a Rust UI layer. It is a Ratatui widget/renderer backed by `libghostty-vt`.

However, it is not a complete PTY/shell terminal. Its `portable-pty` dependency appears dev/example-scoped, not core runtime. It may be useful as a reference or spike target, but it is not the final embedded desktop terminal solution by itself.

### Cortex Code resume behavior

Cortex Code confirms the CLI adapter model Reverie needs.

CLI usage:

```sh
cortex
cortex --resume [session-id]
cortex --model <model>
cortex --yolo
```

Session behavior:

- session IDs are UUIDs generated by Cortex
- saved under `~/.cortex/sessions/{sessionId}`
- each session stores `history.json`, `meta.json`, and optionally `observations.json`
- metadata includes `id`, `mode`, `provider`, `model`, `cwd`, timestamps, context token count, usage, and compaction strategy
- `--resume` without an ID resumes the most recently updated session
- `--resume <id>` resumes a specific session
- `/resume` scopes by project root; `/resume-all` lists all sessions
- `--yolo` maps to auto-approve mode

Reverie should store its own mapping from Reverie session tabs to CLI-native session IDs. For Cortex, that means persisting the Cortex UUID and launching with `cortex --resume <id>` when restoring.

## Downstream implementation scan

Craig pointed to Mitchell Hashimoto's posts as a lead for finding real projects using Ghostty/libghostty. Direct X access was blocked in this environment, but the public Ghostty/GitHub ecosystem gives strong signal.

Useful upstream context:

- Mitchell's `libghostty is coming` article frames `libghostty` as a family of embeddable terminal libraries, beginning with `libghostty-vt`: a zero-dependency VT/parser/state core with Ghostty's Unicode, parsing, fuzzing, low-memory, and protocol work.
- The intended API shape is a clean C API for broad embedding, with future pieces potentially covering input handling, GPU rendering, GTK widgets, Swift frameworks, etc.
- The stability posture remains alpha/pre-1.0: good enough to prototype seriously, not yet something to treat as a boring stable platform.

Relevant open-source projects and what they imply for Reverie:

- `ghostty-org/ghostling`: minimal C terminal emulator using `libghostty` plus Raylib. Confirms the intended split: Ghostty owns terminal correctness/state; the host app owns PTY, input translation, renderer, selection, and windowing.
- `Uzaaft/libghostty-rs`: Rust safe wrappers over `libghostty-vt` with a Rust `ghostling_rs` example using `macroquad`. This is the most direct Rust/Tauri-adjacent path, but still pre-1.0 and source-pinned.
- `wiedymi/restty`: web terminal using `libghostty-vt` in WASM plus WebGPU/WebGL2 and a dedicated text-shaping pipeline. Important lesson: terminal quality needs a real render pipeline, not DOM rows as a long-term strategy.
- `philipp-spiess/electron-libghostty`, `coder/ghostty-web`, `davidmat/onyx-shell`, `0xBigBoss/vscode-bootty`, and Obsidian/JupyterLab integrations: useful reference class for Electron/WebView-style embedding. They suggest a possible Tauri-compatible direction: Ghostty VT/WASM or native bridge behind a canvas/WebGPU renderer.
- `montanaflynn/headless-terminal` and `mwunsch/termscope`: headless `libghostty-vt` tools for deterministic TUI automation, snapshots, waits, and JSON/HTML/PNG/SVG output. These are highly relevant for Reverie's own terminal tests and session snapshots.
- `rwc9u/emacs-libgterm`: Emacs module using Ghostty VT. Confirms the same bridge pattern across a very different host: host owns UI/PTY; Ghostty owns terminal model. Also exposes expected integration pain around Unicode/cell widths and mouse/input gaps.
- `xuzhougeng/phantty`: Windows-only renderer for `libghostty-vt` using DirectWrite/FreeType/sprite rendering. Useful for Windows packaging/rendering concerns, but not a portable frontend.
- Swift/macOS projects such as Muxy, macterm, TermBridgeKit, and GhosttyKit wrappers are useful evidence that native host apps are betting on Ghostty as a core, but they are less directly portable to Tauri.

Implications:

1. The ecosystem validates the architecture Reverie is already moving toward: `PTY runtime -> Ghostty terminal state -> render frame/events -> host renderer`.
2. There is no obvious mature, production-ready Tauri widget to adopt whole. The winning path is likely either a native Rust bridge plus custom canvas/WebGPU renderer, or a Ghostty-WASM/web renderer path if it meets performance and input fidelity requirements.
3. The next spike should not merely prove more terminal correctness; it should prove the Tauri-visible render surface under load. The key risk is frontend drawing/text shaping, not VT parsing.
4. Reverie should borrow headless-terminal/termscope ideas for testability: deterministic waits, snapshots, serialized terminal state, and reproducible TUI assertions.
5. Cross-platform remains the packaging risk: macOS is now promising; Windows/Linux need explicit proof because Zig/source-pinned Ghostty builds and renderer backends differ by platform.

## Recommended implementation shape

### Terminal backend abstraction

Create a Rust-side terminal service boundary roughly shaped like:

```rust
trait TerminalBackend {
    fn spawn(&mut self, spec: TerminalSpawnSpec) -> Result<TerminalId>;
    fn write_input(&mut self, terminal_id: TerminalId, bytes: &[u8]) -> Result<()>;
    fn resize(&mut self, terminal_id: TerminalId, cols: u16, rows: u16) -> Result<()>;
    fn snapshot(&self, terminal_id: TerminalId) -> Result<TerminalSnapshot>;
    fn subscribe(&self, terminal_id: TerminalId) -> TerminalEventStream;
    fn terminate(&mut self, terminal_id: TerminalId) -> Result<()>;
}
```

Keep Reverie's product/session model above this layer:

- Project
- Focus
- Session
- Agent CLI adapter
- Native CLI session metadata
- Terminal process instance

### CLI adapter abstraction

Each supported CLI should define:

- executable detection
- new-session command
- resume-session command
- yolo/dangerous flags
- native session ID extraction strategy
- supported restore behavior

Initial adapters:

- Claude Code
- Codex CLI
- Cortex Code

### Ghostty spike criteria

Before committing the main app to Ghostty-backed rendering, prove:

1. We can build `libghostty-vt` bindings reliably on macOS from a Tauri/Rust workspace.
2. We can spawn a PTY process and feed output into Ghostty VT state.
3. We can render a visible terminal surface in a Tauri window with acceptable performance.
4. We can route keyboard input, paste, mouse, and resize back through the PTY.
5. The same build path has a credible Windows/Linux story.

If these pass, Ghostty-backed terminal becomes v1 path.

If these fail, preserve the same terminal backend API and use a pragmatic backend only as temporary implementation scaffolding while continuing toward Ghostty-quality terminal behavior. A weak terminal is not acceptable v1 product surface.

### Current Ghostty proof status

The isolated `spikes/ghostty-vt-proof` crate now proves several core Ghostty terminal gates on macOS:

- `libghostty-vt` builds successfully from the Reverie repo when Zig `0.15.2` is on `PATH`.
- Static VT bytes render through Ghostty into Reverie's `TerminalFrame` model, including Unicode, truecolor, bold, underline, cursor style, and dirty rows.
- A real non-interactive PTY process can be spawned with `portable-pty`, read with a bounded concurrent reader, fed into Ghostty VT state, and extracted into the same `TerminalFrame` model.
- Interactive PTY input works in the proof: bytes written into `/bin/cat` cross the PTY boundary, echo back through output, and render through Ghostty into Reverie's frame model.
- Resize/reflow works in the proof: a PTY and Ghostty terminal resize from `24x8` to `48x8`, the wrapped text remains intact, and non-empty visual rows reduce from `4` to `2`.
- A long-lived interactive PTY process can stay open across multiple writes: `/bin/cat` remains alive after the first write, accepts a second write before EOF, exits cleanly, and both markers render through Ghostty.
- Sustained high-volume PTY output can drain without deadlock in the proof: a shell loop emits 2,000 marked lines through `portable-pty`, the reader captures all expected markers/line counts, Ghostty ingests the full byte stream, and the final viewport renders the completion marker.

Observed successful proof output:

- static render: styled Unicode terminal content, cursor state, dirty rows, truecolor, bold, and underline
- live PTY output: child exit success, `105` output bytes, PTY size `72x14`, rendered rows include `live pty start`, styled `ghostty colored pty`, Unicode text, and underlined `pty complete`
- interactive input: `71` input bytes and `158` output bytes through PTY echo, rendered through Ghostty at `72x14`
- resize/reflow: `82` output bytes, PTY/Ghostty resize `24x8 -> 48x8`, non-empty rows `4 -> 2`, compacted text preserved across shape change
- long-lived lifecycle: `/bin/cat` remained alive between writes, rendered the first marker before EOF, then rendered both first and second markers after EOF with `136` remaining output bytes at `72x14`
- sustained output/backpressure: child exit success, `150053` output bytes, roughly `1,980` reader events across the run, all `2,000` marked output lines observed, and the final Ghostty viewport at `96x18` rendered lines `1985` through `2000` plus `sustained-output-complete`

This does not yet prove the production renderer or frontend performance path. The next spike gates are Tauri-visible rendering surface/performance, richer keyboard/paste/mouse input, and cross-platform packaging.

### Current Tauri-visible render surface proof status

A minimal Tauri v2 desktop shell now exists under `apps/desktop/src-tauri/`, with a web surface in `apps/desktop/web/`. This is intentionally a render-surface proof rather than the production app shell.

What it currently implements:

- The frontend now runs as the real Vite/React app surface under the monorepo desktop app path, with npm scripts kept at the repository root.
- `apps/desktop/web/terminal-gpu-renderer.ts` owns the current WebGL2-first imperative renderer boundary: backend selection, glyph atlas, cell painting, cursor drawing, overlays, and Canvas 2D fallback.
- `apps/desktop/web/terminal-canvas-renderer.ts` remains the Canvas fallback and synthetic frame fixture helper.
- Reverie's `TerminalCell` model now carries Ghostty's cell width, so wide CJK and emoji cells can be cached, painted, selected, searched, link-hit-tested, highlighted, and cursor-filled without the frontend guessing display geometry. Deep-history search also maps match text offsets back through Ghostty-rendered cells before returning row spans.
- Ghostty-derived partial frames now extract and cross the Tauri boundary as dirty-row diffs rather than full visible-row snapshots. The frontend cache owns unchanged rows, while forced full frames still carry the complete viewport for first paint, resize, scroll jumps, and history replay.
- Deep-history replay now preserves the real terminal surface height when rebuilding Ghostty state. Frontend scroll caches may request larger windows, but Rust stitches those windows from multiple correctly-sized viewport snapshots so Ink-style redraws and Claude Code resume output do not see a different terminal geometry during replay.
- Durable transcript capture now records a fresh-PTY run index per launch. Deep-history replay and search replay each run in a clean Ghostty state, then stitch row windows and match rows into one history coordinate space, which matches resumptions without carrying stale TUI state across launches.
- For local transcripts captured before that run index existed, replay infers Claude Code fresh-launch boundaries from Claude's terminal initialization marker in the first transcript run segment. This keeps legacy Ink redraws from corrupting history by being replayed as one continuous PTY without splitting newer explicit resumed runs a second time.
- History-window requests now return wider backend prefetch bands and keep a small rendered-window cache in the Rust runtime. The terminal still replays with the real viewport height, while adjacent frontend scroll misses can reuse a cached band instead of forcing repeated full transcript replays.
- The production terminal controller now emits paint-loop samples, and runtime session metrics include renderer backend, frame/scroll paint timing, rows/cells painted, WebGL draw calls, GPU upload bytes, and glyph-atlas churn.
- The interaction island now forwards modern SGR 1006 mouse tracking input to the PTY when Ghostty reports mouse tracking mode, while Shift keeps local selection/context-menu behavior available.
- The terminal surface now uses a hidden textarea for focus, paste, and IME composition. The input is positioned on the rendered cursor cell and committed text is forwarded through the same PTY input path as normal key events.
- The frontend now pushes active/background terminal priority into the Rust runtime. Active terminals keep the 16 ms frame cadence; background terminals still process PTY output and transcript capture, but their `terminal_frame` events are throttled to a lower cadence so several live sessions do not all drive 60 FPS WebView events.
- The old JavaScript render proof has been superseded by the React harness plus TypeScript renderer tests.
- The historical proof measured synthetic rendering modes, batch Ghostty frame rendering, and live PTY/Ghostty/Tauri event streaming.
- The Tauri shell exposes a `ghostty_frame_sequence` invoke command that builds real `libghostty-vt` terminal state, extracts Ghostty-derived Reverie `TerminalFrame` snapshots, and returns them across the desktop command boundary.
- The production `start_session` command now owns the live PTY/Ghostty/Tauri event stream. The older standalone live-stream proof command is not part of the current command surface.
- Runtime session exits report render and emit metrics through `record_render_metrics` when the frontend receives the corresponding `terminal_exit` event.
- Root Rust tests pass with `cargo test`; frontend terminal behavior is covered by Vitest renderer/controller tests; the Tauri shell passes `cargo check --manifest-path apps/desktop/src-tauri/Cargo.toml` when Zig `0.15.2` is on `PATH`.
- The browser harness now exercises the terminal interaction path end-to-end against the fixture runtime: launch, tab switching, deterministic terminal output injection, nonblank canvas-pixel verification, selection drag, context menu targets, send-to-input, link detection, IME composition commit, empty-grid menu behavior, multi-session foreground priority, background terminal exit without disarming the foreground terminal, and Claude/Ink-style alternate-screen redraws across coalesced partial frames.
- The browser harness now includes a dedicated WebGL2 renderer budget scenario. It instantiates the production `createTerminalGpuRenderer` in Chrome, verifies the benchmark canvas paints through WebGL2, then paints warmed full-window and dirty-row frames on animation frames with 60 FPS budget gates.
- `npm run dev:terminal-stress` launches the real Tauri shell into the hidden `TauriTerminalStressProof` route. It starts several real PTY sessions, marks one foreground and the rest background, paints through the production WebGL2 renderer on offscreen canvases, and records renderer/runtime metrics from the full Rust to WebView path.

Historical desktop runtime benchmark results:

What this proved:

- Reverie's app boundary can carry Ghostty-derived frame snapshots, not just synthetic JavaScript data.
- A real PTY process can stream bytes into Ghostty VT state, produce incremental `TerminalFrame` updates, and deliver those frames into the Tauri WebView without drops in this controlled proof.
- Canvas dirty-row rendering was credible enough to reach the first production terminal surface, but the long-term renderer direction is now WebGL2-first with Canvas 2D fallback.
- The earlier batch command bridge remains comparatively expensive (`~1s` fetch), but live event streaming changes the shape of the cost: cadence is acceptable, while per-event serialization/emit cost needs continued attention before scaling to heavier output.

What is now covered by focused checks:

- Rust/Ghostty extraction covers Unicode, truecolor, inverse video, input modes, resize reflow, alternate screen state, wide-cell widths, and search span mapping.
- Rust runtime tests cover control-command rejection, viewport scroll commands, active/background terminal frame cadence selection, simulated multi-session output pressure, and fast foreground reactivation when a background terminal has pending output.
- Frontend unit tests cover WebGL2 backend selection/fallback, glyph atlas stats, overlay paint, cursor and block-glyph wide-cell geometry, terminal text-input focus positioning, sparse row caching, lazy history windowing, find span mapping, selection/copy behavior, link actions, context-menu targets, SGR mouse tracking encoding/forwarding, alternate screen cache isolation, resize cache preservation, and renderer context recovery.
- Browser harness smoke covers terminal launch and browser-visible terminal interaction against fixture services, including a canvas-pixel assertion after deterministic output injection, a browser IME composition commit through the hidden textarea, multi-session frontend priority switching, the background-exit foreground-input regression case, and alternate-screen composite selection after coalesced partial redraws.
- Browser harness performance smoke covers the production WebGL2 renderer in Chrome with a 60 FPS paint-budget gate for warmed full-window and dirty-row frame paints.

What it still does not prove:

- Full screen-reader accessibility, native IME candidate-window quality across macOS input methods, or full terminal mouse protocol fidelity beyond the modern SGR 1006 path.
- Heavy long-running concurrent sessions with multiple active-output background sessions through the full Rust/Tauri event pipeline, pathological TUI redraw rates, or sustained real-user scrolling under worst-case output across several live sessions.
- Windows/Linux packaging.

Current implementation direction: keep libghostty-vt on the Rust side for terminal emulation correctness, keep React out of terminal cell reconciliation, and let the frontend own an imperative WebGL2 renderer plus a sparse row cache for scroll, selection, find overlays, and lazy history windows. WebGPU stays behind the same renderer boundary until Tauri's WebView runtime supports it reliably.

## Implementation boundaries

The terminal work should be split into five seams so Reverie can move forward without binding the whole app to a still-unproven renderer path.

### 1. Product domain

Owns durable user concepts and should have no Ghostty, PTY, or process-specific dependencies.

Core objects:

- `Workspace`: local Reverie home plus user settings, including default dangerous-mode preference.
- `Project`: optional folder-backed context. Must not require git.
- `Focus`: user-defined masthead/topic inside a project or the general workspace.
- `Session`: a resumable agent tab attached to a focus.
- `AgentKind`: Claude Code, Codex CLI, Cortex Code.
- `NativeSessionRef`: CLI-native session identifier plus resume metadata.

### 2. Agent adapter layer

Owns CLI-specific behavior and produces process launch specs. It should not know how terminal rendering works.

Responsibilities:

- detect whether an executable is installed
- describe onboarding/setup status
- build new-session commands
- build resume-session commands
- apply dangerous/yolo flags
- discover or extract native session IDs where possible
- normalize restore capability and failure messages

For Cortex Code, this adapter maps Reverie sessions to Cortex UUIDs and restores with `cortex --resume <session-id>`.

### 3. PTY/process runtime

Owns cross-platform process lifecycle and byte streams. It should be usable with any renderer.

Responsibilities:

- spawn process in selected cwd
- attach PTY
- emit output bytes/events
- accept input bytes
- resize cols/rows
- terminate/cleanup process
- report exit status

Likely Rust crates to evaluate here include `portable-pty` and platform-specific alternatives if needed.

### 4. Terminal renderer backend

Owns terminal emulation/rendering and receives bytes from the PTY runtime.

Candidate backends:

- `GhosttyBackend`: target-quality path backed by `libghostty-vt` / Rust bindings if the spike passes.
- `FallbackBackend`: pragmatic v1 path if Ghostty integration requires too much custom surface area.

The UI should talk to this through a stable event/snapshot interface, not Ghostty-specific APIs.

### 5. Tauri UI shell

Owns visible UX: onboarding, project/focus/session navigation, tabs, terminal viewport, settings, and restore affordances.

The UI should know:

- which sessions exist
- which agent kind each session uses
- whether the terminal is running, exited, or restorable
- terminal snapshots/events needed for display

The UI should not know:

- Ghostty internals
- raw PTY management details
- CLI-specific resume flag construction

### First implementation order

1. Define the product/domain model and persistence schema.
2. Define CLI adapter traits and implement Cortex first because source behavior is known.
3. Implement executable detection for Claude Code, Codex CLI, and Cortex Code.
4. Implement PTY/process runtime with a minimal renderer path.
5. Run Ghostty spike in isolation against the same PTY/event contracts.
6. Wire the chosen renderer behind the stable terminal backend boundary.
7. Build restore UX only after native session ID capture/resume is reliable for each adapter.

## Current recommendation

Do not block Reverie's product/data model on Ghostty integration.

Proceed in two tracks:

- Product/app foundation: Tauri v2, local database/config, Project → Focus → Session model, CLI detection/adapters, onboarding, session restore metadata.
- Terminal spike: isolated proof-of-concept for Ghostty-backed terminal rendering.

This lets us protect the core product thesis while still seriously pursuing Ghostty instead of defaulting away from it too early.
